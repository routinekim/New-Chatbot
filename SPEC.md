## Problem Statement

대학교 학생들이 규정집, 매뉴얼, 내부 문서에서 원하는 답을 찾는 데 시간이 너무 걸린다.
"제로 학점 받으면 어떻게 되나요?", "휴학 신청 기한이 언제죠?" 같은 질문을 위해
100페이지짜리 PDF를 직접 뒤져야 한다. LangChain 기반 RAG 시스템으로 이 문제를 해결한다.

## What Makes This Cool

- **한국어 특화**: 일반 영어 임베딩이 아닌 Solar Embedding으로 한국어 규정 특수 표현 이해
- **Hybrid Search**: "학점 3.5 이상" 같은 정확한 숫자/조항 쿼리에서 순수 벡터 검색보다 정확도 높음
- **출처 명시**: 답변과 함께 원문 조항 번호/페이지를 항상 제공 → 학생이 신뢰 가능
- **No auth**: 누구나 바로 접근 가능, 학교 이메일 로그인 없이

## Constraints

- 패키지 매니저: uv
- 프레임워크: LangChain (`langchain>=0.3`, LCEL 파이프라인)
- 언어: Python 3.11+
- 개발 워크플로우: Superpowers TDD — 실패하는 테스트 먼저 작성 → 구현 → green → 다음 기능
- 문서 언어: 한국어 (대학 규정, 매뉴얼)
- MVP: 학생 쿼리 UI — 인증 없음, 공개 접근
- MVP: 관리자 업로드 UI — 부서 담당자용, 최소 인증 필요

## Premises

1. MVP는 순수 벡터 검색에서 시작하지 않고 hybrid search(BM25 + vector)를 처음부터 적용한다. 
   대학 규정 도메인에서 정확한 숫자/조항 쿼리의 정확도가 더 중요하기 때문.
2. 프로덕션 벡터 DB는 Supabase pgvector로 설계한다. ChromaDB는 로컬 개발/테스트 전용.
3. 한국어 대학 문서에는 Upstage Solar Embedding-1-large를 사용한다.

## Approaches Considered

### Approach A: 빠른 MVP → 이후 업그레이드
ChromaDB(dev) + bge-m3-korean + GPT-4o-mini + Streamlit
- Effort: S (2-3일), Completeness: 6/10
- 거부됨: 프로덕션 아키텍처 처음부터 제대로 구축하기로 결정

### Approach B: 프로덕션 아키텍처 (선택됨)
Supabase pgvector + Solar Embedding + Gemini + Hybrid Search + FastAPI
- Effort: M (1주일), Completeness: 9/10
- 선택 이유: 한 번에 제대로, 리팩토링 없이

### Approach C: 로컬 LLM 제로 비용 (EXAONE + Ollama)
- 거부됨: GPU 서버 미확보, 프로드 배포 복잡도

## Recommended Approach

**Supabase pgvector + Upstage Solar Embedding-1-large + Gemini + LangChain Hybrid Search**

### 기술 스택 확정

| 컴포넌트 | 선택 | 이유 |
|---------|------|------|
| LLM | Gemini 2.5 Flash (`gemini-2.5-flash`) | 저렴, 한국어 우수, 긴 컨텍스트 |
| Embedding | Upstage Solar Embedding-1-large | 한국어 RAG 최고 성능 (2025 벤치마크) |
| Vector DB (dev) | ChromaDB (로컬, 세션 간 persist) | 개발 전용 — 프로덕션 데이터 없음 |
| Vector DB (prod) | Supabase pgvector | 스케일, 관리 편의성 |
| 검색 방식 | Hybrid (LangChain EnsembleRetriever: BM25 + vector) | 숫자/조항 정확도 |
| Backend | FastAPI | 비동기, 스트리밍 지원 |
| Framework | LangChain 1.0 (LCEL) | 확정됨 |

### 아키텍처 구조

```
문서 인덱싱 파이프라인:
  담당자 붙여넣기(텍스트/마크다운) → RecursiveCharacterTextSplitter
  (청크 크기: 800자, 오버랩: 200자) → Solar Embedding → Supabase pgvector 저장
  ※ PDF/OCR 파싱 없음 — 관리자 UI에서 HTML→MD 자동 변환 후 직접 입력

쿼리 파이프라인:
  사용자 질문 → Solar Embedding → EnsembleRetriever (BM25 0.5 + vector 0.5)
  → 상위 5개 청크 → Gemini → 출처 포함 답변

테스트 레이어 (TDD):
  - retriever 레이어: VectorStore를 추상 인터페이스로 분리 (dev=ChromaDB, prod=Supabase)
  - generation 레이어: LLM mock으로 deterministic 테스트
  - e2e: pytest fixture로 샘플 문서/쿼리 pair 사용
```

### 청킹 전략

한국어 규정 문서의 특성:
- 조항 단위로 의미가 완결됨 → `RecursiveCharacterTextSplitter`에 한국어 규정 구분자 우선 적용
- 청크 크기: 800자 (한국어 기준), 오버랩: 200자
- 메타데이터: 문서명, **문서종류(doc_type)** 포함 → 출처 표시 + 종류별 필터링용
  - ~~부서(department)~~ 제거됨 — 실제 운영에서 부서 귀속이 모호하고 문서종류로 충분히 분류 가능

```python
# separators 전체 목록 (순서대로 시도, 이전 기본값도 유지)
separators = ["\n제", "\n조", "\n항", "\n\n", "\n", " ", ""]
splitter = RecursiveCharacterTextSplitter(
    separators=separators, chunk_size=800, chunk_overlap=200
)
```

**OCR 여부 결정:**
- 스캔 PDF 없음 → `PyMuPDF` 또는 `pdfplumber`로 충분
- 스캔 PDF 있음 → Upstage Layout Analyzer API 또는 Tesseract 추가, 일정 +2일

### VectorStore 추상화 (TDD 핵심)

```python
# k는 시스템 상수로 취급 (현재 5, 조정 가능)
K_RETRIEVAL = 5

class DocumentRetriever(Protocol):
    def retrieve(self, query: str) -> list[Document]: ...
    # k는 구현체 생성 시 주입, 인터페이스에서 노출하지 않음

# dev: ChromaDB + BM25 EnsembleRetriever (in-memory)
# prod: Supabase pgvector + BM25 EnsembleRetriever (in-memory)
```

> **중요:** BM25는 개발/프로덕션 모두 in-memory로 동작.
> Supabase pgvector가 dense vector를, BM25Retriever가 sparse를 담당.
> MVP 문서 수 5–20개, 예상 청크 ~500개 → 512 MB RAM 이하 문제없음.

### Hybrid Search 가중치

BM25 : vector = 0.5 : 0.5 (초기값). 평가 후 조정 가능.
숫자/조항 정확도 중요 시 BM25 비중을 높임 (0.7:0.3 등).

### 라우팅

```
GET  /                → 학생 쿼리 UI (공개, 인증 없음)
POST /api/query       → RAG 쿼리 API (공개)

GET  /admin           → 관리자 업로드 UI (인증 필요)
POST /admin/ingest    → 문서 등록 + 인덱싱 (인증 필요)
POST /admin/cleanup   → 표 마크다운 정리 (LLM 보조, 인증 필요)
GET  /admin/documents → 업로드된 문서 목록 (인증 필요)
DELETE /admin/documents/{id} → 문서 삭제 + 청크 제거 (인증 필요)
```

**관리자 인증:** MVP는 환경변수 `ADMIN_TOKEN`을 HTTP Basic Auth 비밀번호로 사용.
본격 운영 전에 학교 SSO(SAML/OAuth)로 교체 예정.

### 프론트엔드

- 학생 쿼리 페이지 (`/`): 검색창, 문서종류 필터, 답변 출력(마크다운 렌더링), 출처 목록 — FastAPI + Jinja2
  - 답변: `marked.js`로 마크다운 → HTML 렌더링 (`**굵게**`, 목록, 표 등 포함)
  - 출처: 유사도 60% 이상인 문서만 표시. 80% 이상 초록, 60~79% 노랑 뱃지
  - 유사도 계산: L2 거리 기준 상대 정규화 → `(1 - (score/best - 1) / (threshold - 1)) × 100`
- 관리자 업로드 페이지 (`/admin`): 문서종류 선택 + 문서명 + 텍스트 붙여넣기 폼, 등록 문서 목록(문서종류 표시)/삭제/청크 뷰어
  - ~~부서 선택 드롭다운~~ 제거됨
- 정적 파일: `static/` 디렉토리, `main.py`에서 `/static`으로 마운트
  - `static/turndown.js`, `static/turndown-plugin-gfm.js` — HTML→MD 변환 (관리자 붙여넣기용)
  - `static/marked.min.js` — MD→HTML 렌더링 (학생 쿼리 답변 표시용)
  - CDN 대신 로컬 서빙: 네트워크 환경에 따라 CDN 로드 실패 시 JS 전체가 초기화되지 않음

> **Jinja2 템플릿 주의:** `<div id="...">` 요소를 참조하는 `<script>`는 반드시 해당 요소 **뒤에** 위치해야 함.
> 요소가 스크립트보다 나중에 파싱되면 `getElementById()`가 null을 반환해
> `addEventListener` 호출에서 TypeError 발생. (`#chunk-modal` 모달에서 실제로 발생한 버그)

향후 트래픽/인터랙션 늘어나면 Next.js로 교체.

### 문서 인제스트 (붙여넣기 + HTML→Markdown 자동 변환, Method C)

**웹 UI (담당자용):** `/admin` 페이지에서 부서 선택 + 문서명 입력 + 붙여넣기 → 저장 즉시 자동 인덱싱

```
[담당자가 복사-붙여넣기]
        ↓
[paste 이벤트에서 text/html 또는 text/plain 클립보드 확인]
  — HWP는 text/plain에 HTML 코드를 그대로 담기도 함 → 양쪽 모두 체크
        ↓
[<table> 포함 여부 확인 → DOM API(querySelectorAll)로 셀 추출 → 마크다운 표 변환]
  — Turndown.js는 GFM 플러그인이 <thead> 구조를 요구해 HWP 표에서 실패함
  — 직접 구현한 tableHtmlToMarkdown()이 <td>/<th> 무관하게 처리
        ↓
["✓ 표 구조 자동 변환됨" 배지 + "정리하기" 버튼 표시]
        ↓
[복잡한 병합 셀 → "정리하기" 버튼 → POST /admin/cleanup → Gemini로 정리]
        ↓
[담당자가 확인 후 "등록 및 인덱싱"]
```

**구현 세부:**
- `static/turndown.js`, `static/turndown-plugin-gfm.js` 로컬 서빙 (CDN 의존 제거)
- 표 변환: `tableHtmlToMarkdown()` — DOM 파싱 후 행/열 추출, 첫 행 뒤에 `|---|` 구분자 삽입
- 비표 텍스트: Turndown.js로 일반 HTML→MD 변환 (폴백)

**인제스트 요청 형식:**
```json
POST /admin/ingest
{
  "title": "2025 학생생활규정",
  "doc_type": "규정",
  "content": "제1조 (목적) 이 규정은..."
}
```
- `doc_type` 선택지: `규정` / `지침` / `업무매뉴얼` / `결재공문/기안문` / `계획서/결과보고서` / `회의록` (선택 안 함 허용)

**표 정리 요청 형식:**
```json
POST /admin/cleanup
{ "content": "| 구분 | 내용 |\n| 복잡한 병합 셀... |" }
→ { "content": "| 구분 | 내용 |\n|---|---|\n| A | B |" }
```

**샘플 데이터 (개발용):**
`RAG_샘플데이터/` 폴더에 10개 문서 포함 (규정 2, 지침 1, 업무매뉴얼 1, 기안문 2, 계획서/결과보고서 2, 회의록 1, FAQ 1).
인제스트 완료 — `chroma_db/`에 총 34개 청크 저장됨.

## Dependencies

- Upstage API key (Solar Embedding; 폴백: `bge-m3-korean` HuggingFace 무료)
- Gemini API key (`gemini-2.5-flash`) ← `.env`에 설정 완료
- Supabase 프로젝트 설정:
  1. pgvector extension 활성화
  2. `documents` 테이블 생성 (id, content, metadata, embedding)
  3. `match_documents` SQL 함수 배포 (LangChain SupabaseVectorStore 필수):
     ```sql
     create or replace function match_documents(query_embedding vector, match_count int)
     returns table(id bigint, content text, metadata jsonb, similarity float)
     language plpgsql as $$
     begin
       return query select id, content, metadata,
         1 - (embedding <=> query_embedding) as similarity
       from documents order by embedding <=> query_embedding limit match_count;
     end; $$;
     ```
- BM25 cold-start: 서버 시작 시 Supabase `documents` 테이블에서 청크 로드 → in-memory BM25 재구성. ~500청크, 부팅 < 1초 예상.

## What I noticed about how you think

- "Gemini LLM"을 직접 지정했어요 — 단순히 "OpenAI 써야지"가 아니라 비용 구조를

- 한국어 특화 임베딩이 필요하다는 전제에 바로 동의했어요. 도메인을 제대로 이해하고
